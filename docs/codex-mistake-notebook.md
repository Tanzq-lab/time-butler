# Codex 错题本

这个文件记录 Codex 在帮用户维护 Time Butler 时犯过的细小但会反复影响体验的决策错误。后续 Codex 在添加任务、补录番茄、调整分类、修改数据或实现功能前，必须先参考这里的规则。

## 2026-06-22：任务标签不要用过粗的大领域

### 错误

把任务分类粗略记成 `工作`、`投资`、`社保事务` 这类大领域，导致后续复盘不直观。

这种记录只能说明时间属于哪个生活/业务范围，不能说明这段番茄实际消耗在什么类型的工作上。复盘时很难看出时间花在了开发、修 Bug、测试、配置、资料整理、行政办理还是 Codex 协作。

### 正确做法

- `project` 记录任务归属：属于哪个产品、领域、长期事项。
- `category` 记录工作类型：这段番茄实际在做什么性质的事。
- 不要默认把 `工作`、`投资`、`个人事务` 这类大领域当成分类。
- 如果用户只给了大领域，要优先放到 `project`，再根据任务动作推断更细的 `category`。

### 推荐分类方式

| 任务 | project | category |
|---|---|---|
| 初始化项目 | Time-butler | 项目初始化 |
| 优化项目功能以及打开方式 | Time-butler | 功能优化 |
| 处理 Codex 添加任务问题 | Time-butler | Codex协作 |
| 测试 Codex 新增任务是否成功 | Time-butler | 测试验证 |
| 修复 Time-butler 应用的 BUG | Time-butler | Bug修复 |
| 优化投资Agent工作流 | 投资Agent | 工作流优化 |
| 申请失业补助金 | 个人事务 | 行政办理 |

### 下次执行规则

当用户让 Codex 添加、补录或修改任务时，先判断两个问题：

1. 这个任务属于哪个项目或领域？
2. 这个番茄实际是哪种工作类型？

只有第二个答案才应该进入 `category`。如果分类看起来过粗，先细化为动作/劳动类型，再写入数据库和日志。

## 2026-06-22：预估偏差不只记录超时

### 错误

只在实际番茄数超过预估时要求复盘，忽略提前完成的任务。

提前完成同样说明预估规则可能偏大、任务粒度可能过小，或者任务比描述更简单。如果不记录原因，后续制定类似任务时还是会把时间设得过宽。

### 正确做法

- 任务完成时比较 `actualPomos` 和 `estimatedPomos`。
- 只要两者不一致，就要求写复盘原因。
- `actualPomos > estimatedPomos`：记录为什么晚点做完。
- `actualPomos < estimatedPomos`：记录为什么提前做完。
- 复盘原因要写入 `../time-butler-data/data/pomodoro-estimation-log.jsonl` 的 completion 事件，作为后续预估参考。

## 2026-06-22：不要混淆番茄到点和任务完成

### 错误

把“番茄自然到点”和“用户主动点击完成任务”混成同一个复盘流程，导致计时器到点后弹出不该出现的任务预估复盘要求。

### 正确做法

- 番茄自然到点：自动结束当前番茄，记录 1 个完成番茄，按“自动开始休息”设置进入休息或休息待开始。
- 番茄自然到点：不进入超时计时。
- 手动提前点击当前番茄的“完成”：结束本轮但不算完成番茄，不增加任务进度，回到专注待开始。
- 任务完成复盘：只发生在用户主动点击“完成任务”时。
- 任务完成时实际番茄数等于预估：直接完成，不弹复盘。
- 任务完成时实际番茄数不等于预估：必须填写复盘原因。

## 2026-06-22：外部写库后不要忽略 UI 内存状态

### 错误

Codex 直接把任务写进 `Time-butler.db` 后，就认为任务已经交给 Time-butler；但前端任务列表只在初始化时加载一次任务，用户已经打开 UI 时不会自动感知外部 SQLite 变更。

结果是数据库里有新任务，用户界面里暂时看不到，体验上等同于“任务没添加成功”。

### 正确做法

- 直接写 SQLite 后，要同时确认 UI 是否会重新加载任务。
- 任务页应在进入页面、窗口重新获得焦点、页面从后台回到前台时重新执行 `loadTasks()`。
- 向用户说明未来排程任务显示在 `稍后提醒` 分区，不会出现在 `进行中` 分区。
- 如果 UI 已经打开且没有自动热更新，提示用户刷新窗口或重新进入任务页。

## 2026-07-07：复盘后不要等用户提醒才看 App 建议

### 错误

用户要求日报复盘后，如果发现 Time Butler 需要优化，Codex 应该自己优化并提交。

但第一次执行时，Codex 只重新生成了日报，没有主动检查日报手写内容和 AI 复盘中的 App 建议，也没有检查 App 里对应的可优化点。用户提醒后才发现：

- `2026-07-06` 日报手写内容已经明确说“添加任务的标签越来越多、不直观，希望 Codex 后台自动加标签”。
- 时间计划工作台也存在外部脚本写入后 UI 内存不自动刷新的体验问题。

这说明 Codex 把“复盘后自我优化”理解成可选收尾，而不是必须执行的检查清单。

### 正确做法

每次生成日报、周报、月报，或用户要求“根据复盘优化 App”时，必须主动做下面的检查：

1. 读取本文件 `docs/codex-mistake-notebook.md`。
2. 检索目标日报 / 周报 / 月报内容中的 App 建议关键词，例如 `App`、`APP`、`Time Butler`、`时间管家`、`标签`、`不直观`、`优化`、`建议`、`刷新`、`自动`。
3. 区分两类建议：
   - 可以安全小步优化的 App / 脚本 / SKILL 问题：直接实现、验证、只提交相关文件。
   - 涉及产品判断、历史数据批量修改、分类策略大改或超过 4 个番茄的问题：写入复盘第 10 节，说明为什么暂不自动处理。
4. 如果用户指出 Codex 漏做了自我优化或纠错，必须先更新这个错题本和相关 SKILL / 脚本规则，再继续实现功能。
5. 最终回复要明确说：扫描到了哪些建议、自动处理了哪些、哪些没有处理以及原因。

### 下次执行规则

不要只写“复盘后系统优化：无”。

在写“无”之前，必须能回答：

- 我有没有读用户手写日报？
- 我有没有搜索近期记录里的 App 建议？
- 我有没有检查对应 App 代码？
- 如果没自动处理，是因为大、危险、需要用户判断，还是因为我漏看？

只有前三个都做过，且没有小而明确的可验证优化点，才能写“今日没有发现适合自动处理的 Time Butler 优化点”。

## 2026-07-08：不要只等用户反馈才发现工具问题

### 错误

只根据用户明确指出的问题修工具，缺少对 Time Butler 自身使用路径的观察。

这样会导致一些反复出现但用户没来得及描述的问题被漏掉，例如：频繁进入某页但没有完成动作、任务反复创建后又删除、计时开始后频繁跳过或放弃、时间页面多次编辑但内容变化很小。

### 正确做法

- Time Butler 需要保留本地产品使用事件，不上传外部服务。
- 埋点只记录路径、动作、实体类型/id、计数、布尔特征和长度变化，不复制任务正文、笔记正文或日报正文。
- 日报、周报、月报和复盘后自我优化时，要把 `app_events` 作为辅助证据读取。
- 不能只凭埋点推断用户心理或需求；要和任务数据、session、completion_review、手写复盘或用户明确反馈互相印证。
- 如果埋点显示某个路径反复出现摩擦，并且修复范围小、可验证、不需要改用户数据，就应主动优化工具体验。

### 下次执行规则

每次做 Time Butler 工具体验优化时，至少回答：

1. 当前有没有可用的 `app_events` 数据？
2. 有没有重复路径、放弃路径、过度编辑、反复删除/归档等信号？
3. 这个信号是否能和用户反馈或任务/session 数据互相印证？
4. 是否存在一个不超过 4 个番茄、可验证、不会影响正在计时的小修复？

## 2026-07-08：优化工具前必须先读产品方法论

### 错误

把 Time Butler 优化当成普通工程改动，只看代码和用户一句反馈就直接实现，缺少产品方法论里的需求剥离、产品内核、关键假设、低成本验证和转化率视角。

这会导致 Codex 容易做“看起来能解决问题”的功能，但没有先确认用户路径、真实需求、关键假设和最小可验证改动。

### 正确做法

- 每次优化 Time Butler 作为产品/工具之前，先读 `docs/product-optimization-methodology.md`。
- 按该文档读取 `/Users/amos/AmosTan` 的 `AGENTS.md`、根 `README.md`、`AI_RETRIEVAL_GUIDE.md` 和产品基本功索引。
- 根据问题类型继续读取原始方法笔记，而不是只看 README 摘要。
- 把用户当作用户，把 Codex 当作产品负责人：小而可验证的设计和产品判断由 Codex 自己做，不要求用户替 Codex 设计方案。
- 仍然遵守数据安全边界：不随意改用户任务/session 数据，不引入外部服务，不做无法验证的大改，不因纯数据/规则更新重启 Tauri dev server。

### 下次执行规则

在任何 Time Butler 产品优化前，必须能回答：

1. 我读了哪些 AmosTan 方法论入口和原文？
2. 这个问题属于需求剥离、产品内核、低成本验证、转化率、业务公式中的哪一类？
3. 用户路径和真实需求是什么？
4. 关键假设是什么？
5. 最小可验证改动是什么？
6. 本轮验证结果是什么？

## 2026-07-09：日报不要漏掉任务完成复盘留言

### 错误

生成日报时只把 `completion_review` 当作归纳材料，最终在“今日反思”里只引用少数重点句子。

这会让用户读日报时看不到自己当时对任务留下的完整判断，也会削弱后续预估偏差、任务拆分和 AI 协作反思的证据链。

### 正确做法

- “今日反思”必须包含“原始留言”小节。
- 任务层面：逐条列出当天计划或完成任务中非空的 `completion_review`。
- 番茄/session 的 `intention`、`mood`、`notes` 只作为分析证据，不逐条写入“原始留言”。
- 原始留言可以紧凑排版，但不能只摘要、合并或只挑重点。
- 后续“今日复盘”“学到了什么”“需要改进什么”再基于这些原始留言做判断。

### 下次执行规则

生成日报前必须能回答：

1. 今天有多少条非空任务完成复盘？
2. 这些任务完成复盘是否全部出现在“今日反思 > 原始留言”里？
3. session 的 intention、mood、notes 是否只用于分析，而没有逐条铺到原始留言里？
