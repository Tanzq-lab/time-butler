# Time Butler UI 优化手册

这份文档沉淀 2026 年 7 月的全站 UI 改造、响应式收口、日历滚动修复和品牌图标更新。它不是一份通用审美清单，而是后续 AI 修改 Time Butler UI 时必须执行的项目约束、工作流和验收门禁。

## 1. 先记住产品内核

### 用户路径

Time Butler 的高频路径是：

```text
选择任务 → 开始专注 → 查看进度 → 完成/复盘 → 在记录或日历中规划下一步
```

2026 年 7 月 14–16 日的本地聚合 `app_events` 显示，计时和任务是访问最频繁的页面，记录页其次，日历也有稳定使用。用户在本轮改造中提供的截图反馈进一步确认：真正妨碍使用的不是“功能不够多”，而是关键路径中的可读性、滚动、窗口适配和状态表达不可靠。

### 提炼出的需求

用户要的不是“套一个 Notion 模板”，而是一个安静、清晰、有掌控感的时间工作台：

- 白色是品牌主表面，墨黑承担主要操作和必要对比。
- 选择态应清楚但克制，不能成为大面积黑块。
- 计时、任务和记录必须在真实 Tauri 窗口中立即可用。
- 美感不能以牺牲文字对比度、滚动、键盘操作或数据安全为代价。
- 品牌图标要有独立设计价值，不能拿临时 SVG、emoji 或普通组件图标充数。

### 关键假设

统一的黑白编辑器语言、可靠的响应式布局和明确的交互反馈，会降低关键路径中的视觉与操作阻力，同时不改变任务和计时语义。

### 最小产品核心

UI 优化优先级固定为：

1. 计时、任务、记录三条高频路径可完成。
2. 导航、标题栏、滚动和弹窗等共享外壳可靠。
3. 视觉语义和可访问性统一。
4. 日历、分析、设置和引导按同一系统收口。
5. 品牌图标作为独立资产处理。

不要把新功能、数据结构调整或计时规则修改夹带进视觉改造。

### 风险边界

- 不修改 SQLite schema、任务/session 数据结构、计时引擎、完成规则或隐私策略。
- 视觉验证使用 E2E mock 数据；不向真实任务数据库写测试数据。
- 源码热更新可能刷新界面。改前端或 Tauri 壳层前，要提醒用户并确认计时状态可恢复。
- 纯 UI 改造不新增远程字体、图片服务、埋点或依赖，除非用户明确授权。

## 2. 本轮过程与教训

### 第一阶段：全站黑白改造

目标是把原有暖色品牌改为 Notion 式黑白系统，并统一按钮、表面、导航、弹窗和页面层级。这个方向成立，但第一轮一次触及 80 多个文件，在真实 Tauri 截图出现之前扩得过快。

后续 AI 必须先完成一个“代表屏验证”：优先选择计时页，同时覆盖桌面、320px 最小宽度、深色和一个选择态。代表屏通过用户任务、视觉和可访问性检查后，才把设计系统扩展到全站。

### 第二阶段：真实窗口自检

用户截图暴露了浏览器预览和静态代码审查没有发现的问题：

| 现场问题 | 系统根因 | 固化规则 |
| --- | --- | --- |
| macOS 红黄绿按钮与品牌标题重叠 | Tauri 使用 `titleBarStyle: Overlay`，侧栏未预留原生标题栏 | 侧栏和主内容顶部都保留至少 32px 拖拽安全区，并在真实 Tauri 窗口检查 |
| 已选任务变成一团黑，文字不可读 | 把“品牌黑”误用为普通选择态，文字、图标和背景颜色组合失效 | 普通选择态用浅灰/深灰表面、边框、勾选和字重；黑色只给主操作；对比度自动测试 ≥ 4.5:1 |
| 320px 下设置入口或操作被裁切 | 只按常见手机宽度设计，没有验证 Tauri 的最小窗口 | 支持矩阵必须包含 320×700，并检测所有按钮、链接和表单控件是否越界 |
| 手机记录页首屏只看到页面树 | 把桌面信息架构直接纵向堆叠到手机 | 1024px 以下页面树进入底部抽屉，首屏直接展示当前文档和编辑器 |
| 小字、说明和图标发灰 | muted 色被用于重要信息 | 导航、标签、说明和统计标签至少满足 4.5:1；低对比色只用于禁用或装饰 |
| 手机图标按钮没有可访问名称 | 视觉图标被当成完整语义 | 所有无文本按钮必须有 `aria-label`，菜单和弹窗按角色与可访问名称测试 |

### 第三阶段：滚动与响应式收口

用户在真实日历页无法向下滚动到晚间内容。根因不是数据缺失，而是滚动所有权不清：外壳锁定视口，页面与时间轴又叠加了高度和 `overflow` 规则。

滚动规则：

- `body` 和应用外壳可以锁定视口，但必须明确唯一的页面纵向滚动容器。
- Flex/Grid 子项要参与内部滚动时，祖先链必须有可计算高度，相关子项必须使用 `min-h-0`。
- 日历时间轴是独立的 `role="region"` 滚动区域，页面本身不能产生横向溢出。
- 手机日历的横向滚动只发生在时间网格内部，并自动定位到今天。
- 不能只检查“出现了 22:00”；必须真实滚轮滚动并断言 `scrollTop > 0`，再确认 22:00 进入视口。

### 第四阶段：品牌图标

这轮图标迭代浪费时间的主要原因不是生图能力，而是没有先锁定品牌概念和使用场景。临时 SVG、普通沙漏、emoji 和未经确认的生图方向都不符合“时间管家 + 沙漏 + 数学/数据 + 3D”的品牌要求。

最终采用用户确认的 3D 无限沙漏母图。后续遵守：

1. 当前图标已经定稿，除非用户明确提出，不主动重新设计。
2. 品牌识别变更属于高返工判断。先用文字说明概念、构图、材质、光线和小尺寸表现，得到方向确认后再生成或替换资产。
3. 不用代码临时拼 SVG 冒充品牌图；Lucide 图标只用于应用内功能图标。
4. 一个母图不能直接承担所有尺寸：
   - Dock/App icon：保留 3D 质感和约 13% 光学留白，当前内容缩放约 87%。
   - 应用内品牌图：使用清晰的品牌位图，但控制在侧栏和引导页需要的尺寸。
   - macOS 菜单栏：使用单色、简化、可作为 template image 的符号，不把 3D 彩色图强行压到 16–22px。
5. 更新母图后，运行 Tauri icon 生成流程并逐项检查 macOS、Windows、iOS、Android 产物。文件生成成功不等于 Dock 已更新。

### 第五阶段：Tauri 图标与重复运行实例

出现过“源码已经换图，但 Dock 仍显示旧图”和两个 Time Butler 图标同时存在。根因包括原生构建缓存、生成的 `.app` 被 LaunchServices 命中，以及 dev runtime 与打包 App 同时运行。

后续只能按以下流程操作：

- 日常运行入口只有：

  ```zsh
  open /Users/amos/time-butler/open-time-butler-dev.command
  ```

- 不用 bundle id、应用名、生成 `.app` 路径或 Computer Use `get_app_state` 启动 Time Butler。
- `src-tauri/target/**/bundle/` 是可丢弃构建产物，不是运行版本。
- UI 控制前先验证 1420 端口正在监听，且唯一应用进程是 `target/debug/time-butler`。
- 验证不存在 `*.app/Contents/MacOS/time-butler` 进程。
- 原生图标更改需要 Rust/Tauri 重建；如果 Cargo 没有重新执行图标嵌入，必须在没有活跃计时风险时明确失效原生构建缓存，再通过唯一启动器重启。
- 最后要看真实 Dock、菜单栏和应用内品牌位，而不是只看 `src-tauri/icons` 文件。

## 3. 当前视觉合同

这些值和语义以 [src/styles.css](../src/styles.css) 与共享组件为准；文档用于解释不能破坏的设计意图。

### 色彩

| 语义 | 浅色 | 深色 | 用法 |
| --- | --- | --- | --- |
| 主画布 | `#ffffff` | `#191919` | 品牌主表面 |
| 主表面 | `#ffffff` | `#202020` | 页面、弹窗、输入 |
| 次级表面 | `#f7f7f5` | `#252525` | 选择、导航 hover、分区 |
| 主文字 | `#37352f` | `#f7f7f5` | 标题、正文、主要操作 |
| 次文字 | `#5f5e5a` | `#b3b3af` | 重要说明、标签 |
| 弱文字 | `#72716d` | `#9b9b98` | 非关键辅助信息 |
| 边界 | `#e9e9e7` | `#343433` | 分隔和控件边框 |
| 主操作 | `#2f2f2f` | `#f7f7f5` | 单一主要 CTA |

- 旧 `sahara-*` 仍是兼容别名，必须映射到语义色，不得重新引入旧暖色品牌。
- 分类、完成、成功、提醒和危险等业务含义可以使用颜色。
- 普通“已选择”不是成功状态，不使用绿色；也不是主操作，不使用整块黑色。
- 图标、文字和箭头默认继承 `currentColor`；分类圆点可保留业务颜色。

### 排版、圆角与动效

- 使用系统中文友好的无衬线字体，不加载远程字体。
- 时间和数据使用等宽数字。
- 圆角只使用 6px、10px 和圆形三档。
- 阴影只用于弹窗、浮层和迷你计时器；普通卡片不使用悬浮阴影。
- 动画限制为 140–200ms 的 `opacity`、`transform` 或必要的颜色变化。
- 不使用 `transition-all`、hover 缩放和无意义装饰动画。
- 必须支持 `prefers-reduced-motion`。

### 交互层级

- 每个页面只有一个明确的主要操作。计时页的“开始专注/休息”使用黑白主按钮；绿色只表示完成或成功。
- 导航使用语义化 `NavLink` 和当前页状态。
- 选择控件使用 `aria-pressed`、勾选、边框或字重表达状态。
- 危险操作必须使用应用内确认框，不能使用 `prompt`、`alert` 或 `confirm`。
- 弹窗必须有 `role="dialog"`、焦点锁定与恢复、Escape、遮罩关闭和明确名称。
- 桌面 hover 才出现的操作，在键盘 focus 时也必须出现；手机端收敛到有名称的“更多操作”。

### 响应式外壳

- ≥1024px：完整侧栏。
- 768–1023px：默认 64px 图标侧栏。
- <768px：固定五项底部导航——计时、任务、记录、日历、更多。
- 分析、设置和帮助进入“更多”底部菜单；位于这些页面时，“更多”显示当前态。
- Tauri Overlay 标题栏的侧栏和主内容都保留至少 32px 顶部安全区。
- 支持的最小窗口是 320×700，任何按钮、链接、输入框和菜单都不能越界。

## 4. 后续 AI 的标准工作流

### Gate 0：先读约束

1. 读 `AGENTS.md`。
2. 读 `docs/product-optimization-methodology.md`，按要求查询并回读 AmosTan 原始方法论。
3. 读本手册。
4. 检查 `git status`，保护用户已有改动。
5. 确认是否涉及前端热刷新、Tauri 重启、真实数据或品牌资产。

### Gate 1：写清最小假设

实施前至少写出：

- 用户正在完成哪条路径。
- 截图、反馈、`app_events` 或代码暴露了什么摩擦。
- 这个摩擦背后的真实需要是什么。
- 哪个关键假设需要验证。
- 本轮最小改动是什么。
- 明确不改什么。
- 如何验证，什么结果算失败。

“让全站更好看”不是可执行假设。应改写为类似：“计时页普通选择态改成中性表面后，浅色和深色的任务名称都可读，且用户仍能一眼识别当前任务。”

### Gate 2：先做代表屏

设计方向变化较大时：

1. 先改共享 token 和必要组件。
2. 只在计时页或用户正在反馈的页面完成一张代表屏。
3. 同时看桌面、320px、深色、键盘焦点和长文本。
4. 在真实 Tauri 窗口检查标题栏、滚动和原生行为。
5. 代表屏成立后再扩展到其他页面。

不要把“代码可复用”误解成“应该一次性修改全站”。先用最小页面验证设计判断，再复用代码。

### Gate 3：按系统扩展

扩展顺序建议：

1. token、Button、Surface、PageHeader、Dialog。
2. MainLayout、侧栏、移动导航、滚动容器。
3. 计时、任务、记录。
4. 日历、分析、设置、引导。
5. 空状态、加载状态、长文本、深色和 reduced-motion。

每触及一个页面，都要检查：首屏任务、主要操作、选择态、空状态、长文本、滚动、键盘和最小窗口。

### Gate 4：视觉与交互验证

必须检查以下视口：

```text
1280×720
1268×768
1024×768
768×700
390×844
320×700
```

核心页面额外检查浅色、深色和 reduced-motion。视觉验收不能只靠“没有横向溢出”，还要检查：

- macOS 窗口按钮是否压住品牌或控件。
- 主按钮是否唯一，选择态是否仍然可读。
- 小字和重要标签对比度是否 ≥4.5:1。
- 长任务名是否省略但可通过 title/可访问名称获知完整内容。
- 任务、记录和日历是否能滚动到真实末尾。
- 手机首屏是否直接呈现当前核心任务。
- hover 与 keyboard focus 是否提供相同行为。
- 弹窗关闭后焦点是否回到触发按钮。

现有响应式与可访问性回归集中在 [e2e/accessibility.spec.ts](../e2e/accessibility.spec.ts)。新增现场问题时，先补一个能失败的回归，再修代码。

### Gate 5：工程验证

UI 改动完成后至少运行：

```zsh
npm test
npm run build
npm run test:e2e
cd src-tauri && cargo check
git diff --check
```

若只改文档，可按风险缩小验证范围。若触及 Tauri 原生窗口、图标、菜单栏或通知，必须额外检查真实运行窗口；浏览器 E2E 不能替代原生验收。

最后使用可用的 Web Interface Guidelines 审计所有触及的 UI 文件，重点检查语义、焦点、对比度、滚动、触控目标和 reduced-motion。

### Gate 6：交付与 Git

- 列出改了什么、没改什么、验证结果和已知边界。
- 不用“测试通过”替代真实视觉结论。
- 提交前检查二进制资产、生成文件和 `Cargo.lock` 是否都属于本次范围。
- 品牌图标、Tauri 生成图标和代码引用必须在同一个有意图的提交中。
- 每次完成仓库内修改后都要提交 Git，不需要等待用户再次提醒。
- 提交前先确认工作树和 staged diff，只暂存本次范围；不要夹带用户原有或其他任务的改动。
- 提交失败时不能把任务报告为完成；除非用户明确要求，不自动 push、建 PR 或改写历史。

## 5. 禁止事项

- 不从“一套好看的全站 UI”直接开始改几十个页面。
- 不把黑色当作所有 active/selected 状态的默认答案。
- 不用低对比灰色承载重要说明、标签或当前状态。
- 不在没有约束高度和 `min-h-0` 的祖先链中随意添加 `overflow-auto`。
- 不以 390px 通过代替 320px 最小窗口验收。
- 不只在浏览器检查 Tauri Overlay、Dock、菜单栏或窗口行为。
- 不用 emoji、临时 SVG 或普通功能图标替代已经确认的品牌图。
- 不在未确认艺术方向时反复生图。
- 不启动、安装或验证生成的 `.app`；只使用唯一 dev 启动器。
- 不为了 UI 测试写入真实任务/session 数据。
- 不因视觉改造改变任务、计时或完成规则。

## 6. Definition of Done

只有同时满足以下条件，UI 优化才算完成：

- [ ] 用户路径、需要、假设、最小改动、验证和风险边界已写清。
- [ ] 高频路径的主要操作和状态表达没有改变语义。
- [ ] 白色品牌画布、黑白主操作和语义色边界保持一致。
- [ ] 浅色/深色重要文字与选择态对比度达标。
- [ ] 320×700 到桌面矩阵无横向页面溢出或控件裁切。
- [ ] 长页面和内部时间轴都能滚动到末尾。
- [ ] 键盘能完成主要操作，弹窗焦点闭环成立。
- [ ] reduced-motion 有效。
- [ ] 单元测试、构建、E2E、Cargo 检查和 diff 检查按风险通过。
- [ ] 真实 Tauri 标题栏、Dock、菜单栏和唯一运行实例已按改动范围检查。
- [ ] 未触碰真实用户数据、计时语义和隐私边界。
- [ ] 本次范围已单独提交 Git，未夹带无关改动。
